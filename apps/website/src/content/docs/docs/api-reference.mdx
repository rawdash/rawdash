All endpoints listed here are the canonical Rawdash wire contract — served by
`@rawdash/hono`'s router factories (or any other framework adapter wrapping
`@rawdash/server`'s pure handlers). The examples below assume the default base
path `/`.

## Authentication

Rawdash does not apply authentication by default. Add auth as Hono `before`
middleware on each router factory, or place an authenticating reverse proxy in
front of the server.

For Rawdash Cloud, pass your API key as a `Bearer` token:

```
Authorization: Bearer <api-key>
```

## Health

### `GET /health`

Liveness probe. Returns synchronously with no storage access — safe to call
from uptime checks, Kubernetes probes, etc.

**Response**

```json
{ "status": "ok" }
```

| Field    | Type        | Description    |
| -------- | ----------- | -------------- |
| `status` | `"ok"`      | Always `"ok"`. |

For sync state (last sync, in-flight progress, errors) see `GET /sync/state`.

## Sync

### `GET /sync/state`

Returns the current sync lifecycle state.

**Response**

```json
{
  "status": "succeeded",
  "queuedAt": null,
  "startedAt": null,
  "lastSyncAt": "2026-04-25T10:00:00.000Z",
  "lastError": null
}
```

| Field        | Type                                                                | Description                                          |
| ------------ | ------------------------------------------------------------------- | ---------------------------------------------------- |
| `status`     | `"idle" \| "queued" \| "running" \| "succeeded" \| "failed"`        | Current sync lifecycle state                         |
| `queuedAt`   | ISO 8601 string \| `null`                                           | When the current run was enqueued                    |
| `startedAt`  | ISO 8601 string \| `null`                                           | When the current run started executing               |
| `lastSyncAt` | ISO 8601 string \| `null`                                           | Timestamp of the last successful sync                |
| `lastError`  | string \| `null`                                                    | Last sync error message, if `status` is `"failed"`   |

State transitions: `idle → (queued →) running → succeeded` on success, or
`running → failed` on error. Terminal states (`succeeded`, `failed`, `idle`)
transition back to `queued`/`running` when a new sync is triggered.

### `POST /sync`

Triggers a sync and returns immediately. The sync runs in the background.

<Aside type="caution">
  This endpoint never blocks waiting for the sync to finish. To know when the
  sync completes, poll `GET /sync/state` until `status` is one of `succeeded`,
  `failed`, or `idle`. The `@rawdash/sdk-client.ensureFresh()` helper does this
  for you.
</Aside>

**Response**

```json
{ "queued": true }
```

Returns `{ "queued": false }` if a sync is already in flight (`status` is
`queued` or `running`).

## Config

### `POST /config/validate`

Statically validates a config's widget metrics against the connector registry's
declared resources — the same check the cloud web UI runs before saving a
config. Mounted only when the host wires up `createConfigValidateRouter`.

**Request body**

A `DashboardConfig` object with `connectors` (array) and `dashboards` (object).

**Response (200)**

```json
{
  "errors": [],
  "warnings": [
    {
      "ref": "Dashboard \"eng\", widget \"revenue\"",
      "severity": "warning",
      "message": "sums \"amount\" which connector \"stripe\" declares in cents ..."
    }
  ]
}
```

| Field      | Type                  | Description                                                       |
| ---------- | --------------------- | ---------------------------------------------------------------- |
| `errors`   | array of issue objects | Validation errors — a non-empty array means the config is invalid |
| `warnings` | array of issue objects | Non-fatal warnings (e.g. currency-unit mismatches)               |

Each issue object has `ref` (string — the dashboard/widget it concerns),
`severity` (`"error" \| "warning"`), and `message` (string).

**Response (400)**

```json
{ "error": "Body must be a config with \"connectors\" and \"dashboards\"" }
```

## Widgets

### `GET /dashboards/:dashboardId/widgets`

Fetch all widget data for a dashboard.

**Path parameters**

| Parameter     | Description                                              |
| ------------- | -------------------------------------------------------- |
| `dashboardId` | Dashboard ID as defined in `defineConfig({ dashboards })` |

**Response**

Returns a JSON object with a `widgets` array:

```json
{
  "widgets": [
    {
      "widgetId": "open_prs",
      "connectorId": "github",
      "data": { "value": 12 },
      "cachedAt": "2026-04-25T10:00:00.000Z",
      "syncState": "fresh",
      "status": "ok",
      "meta": { "connectorStatus": "idle" }
    }
  ]
}
```

| Field          | Type                                                          | Description                                                                       |
| -------------- | ------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `widgetId`     | string                                                        | Widget key from your dashboard config                                             |
| `connectorId`  | string                                                        | ID of the connector that sourced this widget                                      |
| `data`         | object \| `null`                                              | Computed metric result, or `null` if not yet synced. For a multi-connector widget this is `null` unless `aggregate` is set, in which case it holds the merged result |
| `series`       | array                                                         | Optional. Present for multi-connector widgets — one `{ key, connectorId, label, data, ... }` entry per connector |
| `cachedAt`     | ISO 8601 string \| `null`                                     | Timestamp of the last successful sync (when known) for the underlying connector   |
| `syncState`    | `"fresh" \| "stale" \| "unsynced" \| "syncing" \| "failing"`  | Optional. Per-widget sync freshness, derived from connector health                |
| `status`       | `"ok" \| "no_data" \| "error"`                               | Optional. The widget's own compute outcome (see below)                            |
| `errorMessage` | string                                                        | Optional. Present when `status` is `"error"` — the compute or connector error     |
| `meta`         | object                                                        | Optional. Connector-specific metadata about this widget's data                    |
| `syncIntervalSeconds` | number                                                 | Optional. The widget's configured refresh interval, when set                      |
| `format`       | object                                                        | Optional. Resolved display-format hints for the widget (e.g. currency)            |

**`status` vs `syncState`.** `syncState` reflects the *connector's* sync freshness; `status` reflects the *widget's own* compute outcome and is the field to render an error/empty indicator from:

| `status`   | Meaning                                                                                                                                |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `ok`       | The metric resolved against matching rows. A legitimate aggregated `0` (rows existed, summed/counted to zero) is `ok`.                 |
| `no_data`  | The query matched **zero** underlying rows — distinct from a real `0`. Surfaces silently-empty widgets that would otherwise render `0`. |
| `error`    | The metric compute threw, or the connector reported a failure (`error`/`auth_failed`, or any `lastError`). See `errorMessage`.         |

Consumers should render a non-`ok` `status` instead of a misleading value — see the OSS `example-nextjs` app for a reference implementation.

**Multi-connector widgets.** A widget's `metric` accepts either a single metric or an
array of metrics — one per connector — so a single chart can combine data from more than
one connector (e.g. iOS App Store + Google Play downloads). Each metric carries its own
`connectorId`, `name`, `field`, `fn`, and an optional `label` used to name its output
series:

```ts
daily_downloads: {
  kind: 'timeseries',
  title: 'Downloads',
  window: '30d',
  aggregate: { fn: 'sum', label: 'Total' }, // optional server-side merge
  metric: [
    defineMetric({ connector: ios,     shape: 'metric', name: 'app_installs', field: 'value', fn: 'sum', groupBy: { field: 'ts', granularity: 'day' }, label: 'iOS' }),
    defineMetric({ connector: android, shape: 'metric', name: 'installs',     field: 'value', fn: 'sum', groupBy: { field: 'ts', granularity: 'day' }, label: 'Android' }),
  ],
}
```

Resolved multi-connector widgets carry a per-connector `series[]`. By default the
top-level `data` is `null` (render the breakdown from `series`). Add `aggregate: { fn }`
to merge the series server-side into `data`. To merge on the client instead, call
`mergeSeries(widget.series, { fn: 'sum' })` (or `mergeSeriesScalar` for stat widgets),
exported from `@rawdash/core`, `@rawdash/sdk-client`, and `@rawdash/sdk-nextjs`.

A `status` widget's `source` likewise accepts an array of connectors, producing a single
combined health badge (worst-of across the listed connectors).

### `GET /dashboards/:dashboardId/widgets/:widgetId`

Fetch a single widget.

**Response (200)** — a single widget entry (same shape as the list items above).

**Response (404)**

```json
{ "error": "Widget not found", "code": "WIDGET_NOT_FOUND" }
```

Also returns 404 with `code: "DASHBOARD_NOT_FOUND"` if the dashboard doesn't
exist.

## Retention

### `POST /retention/retain`

Runs the configured retention policy once across all connectors.

**Response**

```json
{ "triggered": true }
```

On long-lived runtimes (Node), `mountEngine` also starts a background loop
that calls this on the interval configured by `config.retention.intervalMs`
(default 1 hour). On serverless runtimes, disable the loop with
`mountEngine(config, { startRetention: false })` and invoke this endpoint
from your platform's scheduler instead.
